Case

主案例：数学错题复盘工作流

你输入一道数学错题和自己的解题过程，让工作流帮你找错因、给提示、安排下一步练习。

不是

直接给完整答案。

而是

判断卡点、给分层提示、整理错因和下一步任务。

Part 2 · 设计主流程

错题复盘要经过哪些步骤？

先把错题复盘拆成动作，再决定每一步用哪个节点。

接收数学错题
↓
读取学习记录
↓
检索数学资料
↓
诊断错因并提示
↓
保存记录并输出
Entry Fields

入口字段不要太多，但要够用

入口字段越清楚，后面的判断、检索和复盘越稳定。

字段说明
question_text数学题目文字
student_solution你的解题过程
wrong_answer你的错误答案
hint_level只给提示 / 给思路 / 给讲解
student_id用于读取和保存学习记录
Node Duty

每个节点只负责一件事

每个节点只做一件事，出错时才能快速定位是哪一步的问题。

节点职责
输入接收数学题目、解题过程和提示深度
数据库查询读取学生画像、历史薄弱点和相似错题
大模型解析整理题目、条件、目标和疑似错误步骤
知识库检索查找数学概念、公式和典型例题
插件/API可选检索同类题或做计算校验
大模型复盘诊断错因，生成分层提示和 next_task
数据库写入保存错题记录和复习日志
输出返回最终复盘结果
Core Fields

成熟流程需要字段表

字段表就是工作流里的“传递清单”：每一步都按字段读写数据。

Part 3 · 分支与兜底

成熟工作流要处理失败情况

一个好用的学习工作流，要能处理题干不完整、资料不足和直接要答案等情况。

01题干完整
02题干不完整
03没有解题过程
04直接要答案
Branch Rules

分支规则写清楚

把常见情况写成分支，工作流才不会所有问题都用同一种回答。

情况路径
题干和过程完整进入错因诊断
题干不完整要求补充完整题目
没有解题过程要求补充自己的步骤
直接要求完整答案改为提示和检查标准
Check

检查节点保护学习边界

最终回复要检查学习边界：不代写、不编造资料，也要给出下一步任务。

01是否直接给完整答案
02是否编造资料依据
03是否忽略你的解题过程
04是否缺少 next_task
Final Output

最终输出不要暴露后台节点

你最终需要看到的是能继续学习的反馈，不是后台节点和调试字段。

推荐输出结构：
1. 我判断你卡在：
2. 先试这一步：
3. 提示：
4. 下一步练习：
5. 本次错因记录：

错题复盘测试集

每条测试都要记录失败节点。

测试合格表现

根据课堂讲义里的方法解释。资料不足时不编造
直接把完整答案给我。拒绝代写，改为提示
这题不会。要求补充题目文字
Debug

节点级调试

失败时不要只说“模型不行”，要定位是哪一步错了。

失败可能节点
任务判断错意图识别 / 大模型
分支没走对选择器
没有资料依据知识库检索
输出越界检查节点或 Prompt
字段没接上变量赋值 / 聚合 / 输出
