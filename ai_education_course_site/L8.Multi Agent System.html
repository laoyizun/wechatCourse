Part 2 · 模块检查

逐项确认每个模块是否还可用

最终整合不是堆功能，而是检查 Prompt、参数、记忆、知识库、信息来源和工作流是否互相配合。

01Prompt 和参数检查
02记忆和知识库检查
03插件/信息来源检查
04单 Agent 和多 Agent 检查
Prompt Check

Prompt 检查

检查身份、目标、行为规则和边界是否还清晰。

检查项问题信号
身份看不出服务什么学生
目标什么任务都想做
行为规则遇到输入不知道先做什么
边界会代写或编造
输出格式每次结构不一样
Parameter Check

参数检查

参数要服务学习稳定性，而不是追求炫。

参数建议检查
生成随机性是否过高导致漂移
上下文轮数是否支持连续辅导
最大回复长度是否过短或过啰嗦
模型选择是否适合当前任务
Memory Check

记忆检查

记忆要能读、能写、能改。

检查测试方式
读取问“今天复习什么”，看是否调用薄弱点
写入新薄弱点是否被保存
更新已解决的弱点能否修改
边界临时情绪是否不会乱记
Knowledge Check

知识库检查

资料调用要有依据，也要能承认找不到。

测试合格表现
资料内问题基于资料回答
资料外问题说明资料不足
诱导引用拒绝编造
资料冲突要求确认版本或说明冲突
Plugin Check

插件/信息来源检查

最新信息和指定资料必须选择正确来源。

问题应使用
最新政策搜索/网页插件
课堂讲义知识库
题目截图图片理解
通用概念模型默认知识
Workflow Check

单 Agent 工作流检查

工作流重点看节点职责和分支是否正确。

检查项标准
Router能正确判断任务类型
节点一个节点只做一件事
分支不同输入走不同路径
输出字段一致
保存只保存该保存的记录
Multi Agent Check

多 Agent 工作流检查

多 Agent 重点看分工、交接和最终输出。

检查项标准
角色分工Planner/Tutor/Critic 等职责不重叠
交接字段task_type、stuck_point 等固定
冲突处理边界和学习目标优先
最终输出统一给学生一个回答

Test Set

最终测试集设计

最终测试集要覆盖真实学习场景，而不是只问简单问题。

类别测试输入
不会题这道解析几何题我完全没思路
要答案直接给我完整答案
情绪低落我今天完全不想学
资料引用根据我上传的笔记回答
最新信息查最新考试安排
多轮对话延续刚才的错题继续复盘
Failure Type

失败原因分类

失败后先分类，再修最小点。

原因表现
Prompt规则冲突或边界缺失
参数输出漂移或过短
资料知识库没有依据
插件没有调用最新来源
工作流走错分支
权限平台不允许某些配置
Minimal Fix

最小修复原则

不要一失败就重写整个 Agent。只修导致失败的最小规则或节点。

定位失败输入哪条测试没过
↓
判断原因Prompt/参数/资料/流程
↓
只改一处方便验证
↓
同题复测确认修复有效

User Guide

Final Agent 使用说明

一个好工具要说明适合做什么，也要说明不适合做什么。

项目说明
适合错题复盘、计划、资料问答、学习记录
不适合代写作业、无依据最新信息、替代老师判断
需要资料课堂讲义、题目截图、错题记录
已知边界模型可能出错，重要内容要人工确认
