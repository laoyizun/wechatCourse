Part 1 · 问题引入

为什么 Agent 需要长期记忆？

从“AI 怎么又忘了”开始，理解长期记忆不是魔法，而是一套可设计的系统。

01普通对话为什么容易忘
02聊天不会直接改写模型本身
03长期记忆需要外部存储和下次读取
Problem

为什么 AI 总是记不住你？

昨天说过的薄弱点，今天换一轮对话可能就消失了。这个问题不能只靠“请你记住我”。

表面现象

AI 好像很懂你，但下次又忘。

真实原因

普通对话不会自动写入长期存储。

本节目标

设计可读取、可提出更新建议、可测试的记忆结构。

Misunderstanding

误区：聊天不会改写模型本身

模型不会因为你说了几句话，就永久改变参数。长期记忆通常来自外部存储和上下文注入。

普通对话只影响当前上下文
↓
外部存储变量、数据库、记忆摘要
↓
再次读取下次回答前把相关记忆放回上下文
Part 2 · 底层机制

长期记忆系统如何工作？

把“记住我”拆成判断、生成记录、Workflow 写入、下次读取几个步骤，学生才能真正理解系统在做什么。

01先判断信息是否值得长期保存
02区分 Agent 读取和 Workflow 写入
03下次对话前读取相关记忆
04把记忆用于计划、讲题和复盘
Memory System

记忆系统的真实结构

先判断信息是否值得记录，再决定由谁写入，最后规定下次如何读取。

学生输入目标、问题、错因、状态
↓
判断价值是否稳定、是否确认、是否有用
↓
生成记录建议交给 Workflow 写入
↓
下次读取用于计划、讲题、复盘
Variable

变量：保存当前状态

变量适合保存“当前”或“长期稳定”的信息，不适合保存一条条历史记录。

变量适合保存
grade年级或学习阶段
current_subject当前主学科
current_goal近期目标
weak_points已确认薄弱点
preferred_style偏好讲解方式
Database

数据库：保存一条条记录

数据库适合保存学习事件；单 Agent 可读取，新增或修改记录通常交给 Workflow 完成。

字段说明
date记录日期
subject学科
question问题摘要
stuck_point卡点
agent_adviceAgent 建议
next_task下一步任务
status完成状态
Compare

变量 vs 数据库

先分清存储类型，后面 Agent 才不会乱记。

类型特点例子
变量少量、当前、经常读取当前目标、薄弱点
数据库多条、历史、可筛选；由 Workflow 写入更可靠错题记录、复盘日志
Part 3 · 记忆设计

Agent 应该记什么，不该记什么？

会保存不等于会记忆。真正有用的记忆，需要字段清楚、来源可靠、对后续学习有价值。

01设计学生画像字段
02设计学习记录字段
03判断哪些信息值得保存
04避免错误记忆污染后续建议
Profile

学生画像字段设计

学生画像不是越多越好，字段要能服务回答和计划。

字段用途示例
年级控制讲解难度高一
主学科筛选相关记录数学
当前目标生成计划函数单元提分
薄弱点复盘和推荐定义域、单调性
讲解偏好调整输出方式先提示再讲解
Learning Records

学习记录字段设计

记录字段要能支持“下次计划”和“本周复盘”。

字段记录内容
问题学生原始问题或摘要
任务类型讲题/计划/复盘/资料问答
卡点概念/审题/步骤/表达
建议本次 Agent 给出的建议
下一步学生可执行任务
状态未完成/已完成/需复测
Worth Remembering

哪些信息值得记

只有稳定、确认、有后续价值的信息才值得进入长期记忆。

01反复出现的薄弱点
02学生明确确认的目标
03已完成或未完成的任务
04多次测试暴露的错因
05学生偏好的讲解方式
Not Remember

哪些信息不该乱记

乱记比不记更糟，因为错误记忆会污染后续建议。

01临时情绪不要直接记成长期特征
02Agent 猜测的错因不要直接保存
03隐私敏感信息不要保存
04一次性玩笑不要当成偏好
05未经确认的信息先询问
Read Rule

记忆读取规则

不是所有记忆都要每次读。只读取和当前任务相关的信息。

回答前，先读取与当前学科、当前目标、最近薄弱点相关的记忆。
如果记忆与本次问题无关，不要强行提及。
如果记忆可能过时，先向学生确认。
Update Rule

记忆更新规则

什么时候建议写入，比写什么更重要。

只有在以下情况生成记忆更新建议：
1. 学生明确确认目标或薄弱点；
2. 压力测试或多次记录显示同一问题反复出现；
3. 学生完成任务并反馈结果。
不确定时，先问“这个需要保存到你的学习档案吗？”；需要改写数据库时，交给 Workflow 的新增/更新节点处理。
Risk

错误记忆的风险

如果 Agent 错记“你总是不会函数”，它可能之后一直给过低难度建议。

错误记忆后果修复
错记薄弱点计划长期偏离要求确认再保存
错记目标推荐任务不相关允许学生更新目标
错记偏好输出风格不适合保留偏好修改入口

Coze Config

Coze 中的配置位置

本节重点看变量、数据库读取和更新建议；真正改写数据库的动作放到 Workflow 里完成。

变量

保存学生当前画像和偏好。

数据库

保存一条条学习记录。

Prompt 规则

规定何时读取、何时建议更新。

Read vs Write

单 Agent 读取，Workflow 写入

在 Coze 里，Agent 应用可以根据数据库内容回答；但新增、修改、删除数据库记录，需要用 Workflow 的数据库节点更可靠地完成。

动作课堂处理例子
读取数据库单 Agent 应用中配置读取规则根据最近学习记录建议今日任务
生成待保存内容Agent 输出结构化记录待保存：学科、卡点、建议、下一步
改写数据库Workflow 使用 Add data / Update data 节点新增一条学习记录，或更新薄弱点状态
Prompt With Memory

Prompt 如何配合记忆

Prompt 不是替代记忆，而是规定如何使用记忆。

当学生提出学习问题时：
1. 读取当前学科、当前目标和最近薄弱点。
2. 如果相关，结合这些信息给建议。
3. 回答末尾判断是否需要保存本次学习记录，并输出“待保存记录”。
4. 不要承诺自己已经改写数据库；需要写入时交给 Workflow。
Memory Test

记忆压力测试

测试它是否真的能跨对话调用学生画像，并能提出正确的更新建议。

轮次输入合格表现
第 1 轮我最近函数题总是卡在定义域询问是否保存为薄弱点，并生成待保存记录
第 2 轮今天复习什么？优先建议定义域相关任务
第 3 轮我这个弱点已经解决了建议更新或移除薄弱点，不假装已经写库

v3 完成标准
