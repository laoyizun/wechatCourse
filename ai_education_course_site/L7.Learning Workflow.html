Overload

单 Agent 什么时候过载？

当一个 Agent 同时承担太多标准，Prompt 会越来越长，失败也更难定位。

01Prompt 越写越长
02讲解和检查互相冲突
03资料调用越来越复杂
04保存字段越来越多
05失败时不知道是哪一步错
Decision

不是复杂就拆，是职责不同才拆

多 Agent 不是越多越好。只有职责、标准和输出明显不同，拆分才有价值。

不需要拆可以拆
只是换学科规划、讲解、检查职责不同
只是换语气需要互相质检
只是输出格式不同资料调用和最终回答要分开
简单问答记录字段复杂
Part 2 · 从节点到角色

L6 节点如何升级成 L7 角色？

这页是 L6 到 L7 的核心映射：不是凭空增加 Agent，而是把过重节点角色化。

L6 节点L7 角色
Router / PlannerPlanner Agent
ResourceResource Agent
GenerateTutor Agent
CheckCritic Agent
复盘总结 / SaveReviewer / Memory Agent
OutputFinal Response
Team

每个 Agent 是一个岗位

多 Agent 的价值不是“人多”，而是职责清楚。

01Planner：安排任务
02Resource：找资料
03Tutor：讲解
04Critic：质检
05Reviewer / Memory：复盘和记录
Planner / Resource

先决定做什么，再找依据

Planner 不讲题，只判断任务和安排顺序。Resource 不生成最终回答，只负责找到依据。

角色输出字段
Plannertask_type、subject、input_type
Resourceevidence、source_status
Tutor / Critic

一个讲解，一个质检

Tutor 负责分层提示和讲解；Critic 负责检查代写、编造、越界和依据不足。

Tutor

输出 draft_answer、stuck_point、next_task。

Critic

检查风险，必要时打回修改。

Reviewer / Memory

一个复盘，一个记录

Reviewer 总结错因、进步点和下一步建议；Memory 整理结构化学习记录。

角色字段
Reviewerreview_summary、next_task
Memorymemory_update、save_status
Final Response

最终只能有一个声音

多个 Agent 可以协作，但不能让学生看到一堆角色各说各话。

01任务判断
02关键帮助
03下一步行动
04记录说明
Part 3 · 交接与连线

交接字段就是团队工单

多 Agent 之间不要只传一大段自然语言，要用固定字段交接。

01question_summary
02evidence
03draft_answer
04critic_notes
05memory_update
Main Chain

先做主链路，再加分支

第一版多 Agent 不要做成复杂网状结构，先让一条主链路跑通。

Branch

不同任务可以少走一些角色

多 Agent 不是每次全员上场，简单任务要走短路径。

任务建议路径
题目讲解Planner → Resource → Tutor → Critic → Memory → Final
今日计划Planner → Memory → Final
作文修改Planner → Resource → Tutor → Critic → Reviewer → Final
资料不足Resource → Final 提示补充资料
Priority

多 Agent 冲突时听谁的？

冲突优先级要提前写清，否则多个角色会互相拉扯。

边界规则
>
学生输入限制
>
资料依据
>
学习目标 / 表达美观
Part 4 · 测试与风险

多 Agent 要用复杂任务测试

简单问答不需要多 Agent。测试要覆盖规划、资料、讲解、检查、记录多个环节。

测试输入

我下周要函数小测，定义域和单调性都不稳，帮我安排复习，并讲一题。

理想流程

Planner → Resource → Tutor → Critic → Reviewer → Memory → Final

Risk

多 Agent 也会带来成本

拆分是为了稳定和可维护，不是为了显得复杂。

风险控制方式
更慢只在复杂任务使用
重复输出设置 Final Response
交接不清固定字段
角色冲突写清优先级
