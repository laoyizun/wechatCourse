L7 · Multi Agent

多 Agent
工作流

不是再做几个学科 Agent，而是让多个角色分工协作完成复杂学习任务。

PlannerTutorCriticReviewer交接字段
Problem

一个 Agent 为什么越写越乱？

当规划、讲解、检查、复盘都塞进一个 Prompt，角色就容易冲突。

症状

Prompt 很长，回答却不稳定。

原因

职责太多，优先级互相打架。

解决

把明显不同的职责拆开。

Overload

单 Agent 过度复杂的表现

不是所有复杂都要拆，但出现这些信号就该考虑多角色。

01Prompt 太长，学生看不懂
02既要温柔鼓励，又要严格批改
03既规划又讲题又复盘，输出混乱
04一个失败点牵动整段 Prompt
05测试时很难定位问题
Not More Better

多 Agent 不是越多越好

只有职责真的不同，拆分才有意义。

不该拆可以拆
只是不同学科同一套结构换资料即可
只是语气不同一个 Agent 内部规则即可
规划/讲解/批改/复盘职责不同适合拆角色
需要互相检查适合多 Agent 协作
When

什么时候需要多 Agent

任务需要不同角色连续处理时，多 Agent 才有价值。

规划决定做什么
↓
讲解帮助理解
↓
检查发现错误
↓
复盘总结和记录
Roles vs Agents

多角色 vs 多 Agent

可以先在一个 Agent 内模拟多角色，复杂时再拆成多个 Agent。

方式适合情况
一个 Agent 多角色课堂原型、低复杂度
多个 Agent角色职责清楚、需要交接和检查
工作流节点模拟角色更稳定，便于测试
Planner

Planner：负责规划

Planner 不讲题，只负责拆目标和安排任务。

你是 Planner。请根据学生目标、可用时间、薄弱点生成任务计划。不要讲解知识点。输出：任务、时间、完成标准。
Tutor

Tutor：负责讲解

Tutor 不安排长期计划，只负责讲题和分层提示。

你是 Tutor。请根据学生问题和资料进行分层提示。不要代写完整答案。输出：卡点、提示层级、下一步。
Critic

Critic：负责检查答案

Critic 的任务是指出问题，不负责安慰或规划。

你是 Critic。请检查学生答案是否符合题目要求或资料标准。输出：正确点、问题、修改建议。
Reviewer

Reviewer：负责复盘

Reviewer 读取学习记录，总结模式和下一步建议。

你是 Reviewer。请根据学习记录总结高频卡点、进步点和下周最小行动。不要重新讲题。
Memory

Memory：负责整理记录

Memory 不和学生长篇对话，只把学习事件整理为结构化记录。

你是 Memory。请把本次学习事件整理为字段：date、subject、question、stuck_point、next_task、status。
Handoff

交接信息是什么

Agent 之间不能只传一大段自然语言，交接字段要固定。

字段说明
task_type任务类型
subject学科
question问题摘要
stuck_point卡点
evidence资料依据
next_task下一步任务
confidence是否需要人工确认
Order

谁先处理

一般由 Planner 或 Router 先判断任务类型，再分配给对应角色。

Router/Planner判断任务类型
↓
Tutor/Critic处理学习内容
↓
Reviewer总结学习模式
↓
Memory保存结构化记录
Final Output

谁最终输出

多个 Agent 可以协作，但给学生的最终回答要统一口径。

问题

多个 Agent 同时说话，学生会混乱。

做法

由一个 Final Response 节点整合输出。

Conflict

冲突时听谁

角色冲突时，优先级要提前写清。

冲突优先级
Tutor 想直接讲答案，Boundary 不允许边界规则优先
Planner 安排太多任务，学生时间不足学生时间约束优先
Critic 很严格，Reviewer 要鼓励先指出问题，再给可执行改进
Flow

多 Agent 工作流图

Risk

多 Agent 风险

多 Agent 不是免费午餐，复杂度会增加。

风险控制方式
延迟更高只在复杂任务使用
成本更高减少无用角色
输出重复设置最终整合节点
交接不清固定交接字段
v7 Standard

v7 完成标准

能说明为什么拆、拆成谁、怎么交接、怎么输出。

01角色职责清楚
02交接字段固定
03冲突规则明确
04流程图完整
05至少一个复杂场景跑通
